- Formalize commit body standard in a single config place — currently spread across 01_production_git_workflow_standards.md, AGENTS.md, QWEN.md, validate_commit_msg.py, generate_changelog.py. Need a SSoT config (e.g., .vadocs/commit-body.conf.json or pyproject.toml section) that all docs and scripts reference. All contracts should be written to scripts and configs, not prose docs.
- Move BROKEN_LINKS_EXCLUDE_LINK_STRINGS from paths.py to pyproject.toml — check_broken_links.py already follows the pattern of loading changelog-exclude-patterns and valid-types from [tool.commit-convention].
- Add pyproject.toml to the consumers list for external-repos.conf.json — when a repo directory is relocated via manage_external_repos.py, the pyproject.toml exclusion patterns should update alongside .gitignore.
- Release preparation commit body format — agent-generated release commit messages use terse single-line bullets without sub-bullets, which technically pass the hook but don't follow the changelog-rich format expected for release commits. The hook should either enforce sub-bullet detail for release/doc commits, or there should be a separate template for release commits that captures the structured detail needed for changelog generation.
- Enforce script+test co-staging in check_script_suite.py — current implementation only checks that the test file EXISTS (naming convention, check_naming_convention() at line 112), not that both script and test are staged together. AGENTS.md claims "Pre-commit hook enforces the dyad: when modifying a script, the corresponding test must also be staged" but the actual hook has no staged-file validation. ADR-26045 supersedes ADR-26011, and staging enforcement was dropped during simplification. Files: tools/scripts/check_script_suite.py, .pre-commit-config.yaml line 56 (check-script-suite hook). Need to add staged-file validation: if a script in tools/scripts/ is staged, its test in tools/tests/ must also be staged, and vice versa.
- Validate ArchTag values against an allowlist — `validate_commit_msg.py` uses `_ARCHTAG_RE = re.compile(r"^ArchTag:\S+")` which accepts ANY non-whitespace string. No registry of valid tags exists in pyproject.toml. History shows freeform tags: `ADR-26046`, `REFACTOR-MIGRATION`, `GUIDES-RESTRUCTURE` (invented on the spot). Files: tools/scripts/validate_commit_msg.py (line 67 regex, line 147 validate_archtag), pyproject.toml [tool.commit-convention]. Need `archtag-valid-values` list and validation against it.
- Require `options.type` for all markdown files with frontmatter — `check_frontmatter.py` (line 190) emits only a WARNING and silently skips validation for files with frontmatter but no `options.type`. This means any .md file with a YAML block bypasses schema validation entirely. The design should be: all .md files with frontmatter MUST declare `options.type` to be governed; files without a type should fail, not warn-and-skip. Files: tools/scripts/check_frontmatter.py (line 190, validate_parsed_frontmatter), tools/tests/test_check_frontmatter.py.
- Fix `scan_paths()` exclusion logic in `check_frontmatter.py` — uses `any(part in VALIDATION_EXCLUDE_DIRS for part in child.parts)` which does exact-match on individual path segments, but `VALIDATION_EXCLUDE_DIRS` contains composite paths like `'ai_agents/research/ai_coding_agents'`. Individual parts ('ai_agents', 'research', 'ai_coding_agents') aren't in the set, so external repo files leak through validation. Should use `paths.is_excluded()` or substring matching. Files: tools/scripts/check_frontmatter.py:scan_paths() line ~255, tools/scripts/paths.py:is_excluded().
