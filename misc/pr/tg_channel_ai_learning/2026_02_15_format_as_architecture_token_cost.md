**Формат промпта — это архитектурное решение, а не вопрос вкуса**

💡 **Проблема: формат не нейтрален**

Когда вы отправляете системный промпт в LLM, модель видит не "файл в формате JSON" или "файл в формате YAML". Она видит последовательность токенов — и обрабатывает каждый из них с одинаковой вычислительной стоимостью через механизм attention.

Это значит, что кавычки `"`, фигурные скобки `{}`, запятые `,` получают ровно столько же "внимания" трансформера, сколько и ваши реальные инструкции. Каждый структурный символ — это токен, на который модель тратит ресурсы, которые могла бы потратить на понимание ваших команд.

Вопрос не "какой формат красивее", а **какой формат доставляет максимум полезного сигнала на каждый потраченный токен**.

🧠 **Три формата — три разных "мира" для трансформера**

Рассмотрим один и тот же фрагмент промпта в трёх форматах:

**JSON** — формат данных. Каждое значение обёрнуто в кавычки, каждый уровень вложенности отмечен скобками:
```json
"patience_with_attempts": {
  "rule": "Дай ученику две попытки",
  "on_first_miss": "Подскажи с другой стороны"
}
```
Здесь ~22 структурных токена: 14 кавычек, 2 скобки, 4 двоеточия, 2 запятые. В большом промпте на 300 строк **20-30% всех токенов — это шум формата**.

**YAML** — формат конфигураций. Структура выражается отступами, а не символами:
```yaml
patience_with_attempts:
  rule: Дай ученику две попытки
  on_first_miss: Подскажи с другой стороны
```
Структурных токенов — всего 4 двоеточия. Отступы обходятся дёшево: BPE-токенизатор сжимает 4 пробела в один токен.

**Minified JSON** — тот же JSON, но без пробелов и переносов строк, всё в одну строку:
```json
{"patience_with_attempts":{"rule":"Дай ученику две попытки","on_first_miss":"Подскажи с другой стороны"}}
```
Самый дешёвый по токенам — но без единого "ориентира" (переноса строки, отступа), по которому трансформер мог бы разделить инструкции на смысловые блоки.

⚙️ **Реальное измерение: один промпт — четыре формата**

Мы измерили production-промпт (`devops_consultant.json`) через BPE-токенизатор `cl100k_base` (GPT-4):

```
Minified JSON:    2 306 токенов  — самый дешёвый
Standard YAML:    2 482 токенов  — +7.6%
Pretty JSON:      2 734 токенов  — +18.6%
YAML Literal (|): 2 798 токенов  — +21.3%
```

Два неочевидных результата:

✅ **Standard YAML дешевле Pretty JSON на 252 токена (9.2%).** Отступы и двоеточия обходятся дешевле, чем кавычки и скобки — потому что BPE-токенизаторы умеют склеивать ` пробел+слово` в один токен.

✅ **YAML с `|` (literal block) — самый дорогой формат**, даже дороже Pretty JSON. Стиль `|` заставляет каждую строку иметь глубокий отступ, и "налог на пробелы" перевешивает экономию на кавычках.

🔑 **Но дешевизна токенов ≠ качество рассуждений**

Это главный инсайт статьи. Minified JSON экономит 21% токенов — но убирает все структурные ориентиры (переносы строк, отступы), по которым трансформер навигирует в длинных инструкциях.

Standard YAML стоит всего на 176 токенов дороже, но сохраняет **attention-якоря** — визуальную иерархию, которая помогает модели не "потеряться" в промпте на 2 500 токенов. Тесты показывают, что модели лучше следуют инструкциям в YAML, чем в "стене текста" minified JSON.

Есть и второй фактор: **распределение обучающих данных**. JSON в обучающей выборке — это API-ответы, дампы баз, конфиги пакетов. Модель ассоциирует JSON с "данные для парсинга". YAML — это Kubernetes-манифесты, Ansible-плейбуки, CI/CD-пайплайны. Модель ассоциирует YAML с "инструкции для исполнения".

📐 **Практическая рекомендация**

| Задача | Формат | Почему |
|--------|--------|--------|
| Хранение и редактирование промпта | JSON | Строгий синтаксис, валидация, `jq` |
| Отправка промпта в LLM | Standard YAML | Минимум шума + attention-якоря |
| Жёсткий лимит контекстного окна | Minified JSON | Самая низкая стоимость в токенах |
| Код/скрипты внутри промпта | YAML Literal (`\|`) | Сохраняет форматирование кода |
