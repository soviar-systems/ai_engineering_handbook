🧠 **Зачем изучать чужих агентов**

Цель этого исследования — не обзор инструментов «что выбрать». Я изучаю агентные фреймворки, чтобы **создавать своих агентов**. Для этого нужно понимать, как они устроены внутри: как управляют контекстом, как обнаруживают и загружают навыки, как сохраняют стабильность при дрифте LLM.

Систематический анализ семи open-source проектов (Qwen Code, Claude Code, OpenCode, OpenClaude, Aider, KiloCode, Superpowers) дал результат, который перевернул структуру самого репозитория: агент — это не компонент, который можно «добавить» к системе. Агент — это **собранный продукт**, который потребляет все компоненты. Как автомобиль и каталог запчастей: слои — это двигатель, трансмиссия, подвеска, а агент — готовая машина.

Полный анализ — [A-26022: Agents Break the Layer Model](https://soviar-systems.github.io/ai_engineering_handbook/a-26022-agents-break-the-layer-model/) в репозитории.

⚙️ **Управление контекстом: 5 агентов, 5 стратегий**

Каждый из 5 агентов использует принципиально разный подход к управлению историей сессии:

| Агент             | Стратегия                                                               |
|--------------------|-------------------------------------------------------------------|
| Qwen Code   | Буфер автосжатия + `/compress`                     |
| Claude Code | 5-уровневая система                                          |
| OpenCode     | Реактивная компактизация                             |
| Aider               | Асинхронное фоновое суммаризирование |
| KiloCode         | Форк OpenCode                                                    |

Понимание этих различий — практическая необходимость при проектировании собственного агента. [Обзор](https://soviar-systems.github.io/ai_engineering_handbook/overview/) и [таблицы сравнения](https://soviar-systems.github.io/ai_engineering_handbook/comparison/) — в новом разделе `ai_agents/architecture/context_management/`.

🔧 **Код как основная документация**

Принято новое архитектурное решение: **[ADR-26045: Code as Primary Documentation](https://soviar-systems.github.io/ai_engineering_handbook/adr-26045-ai-native-development-code-as-primary-do)**. Каждый исходный файл в экосистеме должен начинаться с contract docstring — ответа на вопрос «Что должен прочитать будущий агент, чтобы безопасно работать в этом файле?»

Это следствие context engineering: когда агенты — основные потребители кода, каждый токен документации конкурирует с кодом за бюджет внимания. Код с устаревшим docstring падает при импорте. Проза-документация, противоречащая коду, проходит CI бесшумно. Contract docstring + тесты — единственный слой, который остаётся корректным «по конструкции».
